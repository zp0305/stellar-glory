# 学习平台网站开发标准手册

> 版本：v2.0  
> 状态：正式版  
> 日期：2026-04-23  
> 用途：指导学习平台网站从立项到迭代的完整开发流程

## 使用说明

本手册按照标准网站开发流程组织章节，每个步骤的主体是**通用标准做法**，参考案例和经验教训以1~2句话简述，控制在每节末尾。

手册的使用方式：按章节顺序阅读，遇到具体问题时查阅对应章节。每个章节的结构为：

```
标题（标准做法主体）
  ↓
做法描述
  ↓
参考案例：（1~2句）
教训：（1~2句）
```

# 第一章：立项与定位

## 1.1 确定产品核心价值

**标准做法：**

在动手之前，回答三个问题：

1. **我们解决什么人的什么问题？** --- 不是功能，是痛点。高中生学物理觉得"公式背了但不会用"，这就是痛点，不是"我们有42个解题模型"。
2. **凭什么是我们？** --- 差异化在哪里。是内容质量、呈现方式、还是价格？三者都想要 → 三者都平庸。
3. **优先级是什么？** --- 第一版只做一个学科的一个模块，做透；不要第一版就覆盖六科。

**交付物**：一页纸的产品定位文档，包含：目标用户、核心问题、差异化、第一批做哪些内容。

**参考案例**：Brilliant 用"learn by doing"一个切入点打透高中生市场；Khan Academy 用"免费+全学科+名师视频"建立规模优势；洋葱学园用3分钟短视频+随堂练习针对中国高中生课后辅导。

**教训**：什么都想做的产品 → 最后什么都没做好（星耀早期想把物理/数学/化学/语文同步上线，第一版只上了物理还是半成品）。

## 1.2 竞品研究方法

**标准做法：**

选择3~5个同类产品，从以下三个维度提取设计原则：

| 维度 | 看什么 | 提取什么 |
|------|--------|----------|
| 产品定位 | 目标用户是谁、解决了什么问题、不做什么 | 差异化方向 |
| 内容结构 | 内容如何分层、每层的粒度是什么 | 分层参考 |
| 技术架构 | 用了什么技术栈、数据怎么存、用户系统怎么设计 | 工程参考 |

**关键原则：不抄界面，抄设计原则。** 界面抄来没有意义，设计原则抄来才能复用。

**研究结果的记录方式**：每个案例一份文档，包含：产品概况、核心设计原则、可借鉴点、不可借鉴点。

**参考案例**：洋葱学园以3分钟短视频为内容粒度，适配中国学生的学习习惯；StudyNotion 以视频课程为内容形态，配合 C2C 电商。

**教训**：研究不足就动手 → 后期发现 KaTeX 字体 CDN 在部分网络下 DNS 失败，方案返工3次。

## 1.3 技术选型

**标准做法：**

技术选型分两步：

**第一步：列出约束条件**
- 团队最熟悉什么？（不熟悉的技术 → 培训成本）
- 内容规模多大？（小型：静态文件足够；中大型：需要数据库）
- 预计用户量多少？（决定是否需要分布式架构）

**第二步：按维度打分评估**

| 候选方案 | 团队熟悉度 | 维护成本 | 扩展性 | 总分 |
|----------|-----------|----------|--------|------|
| React + Vite | 5 | 3 | 4 | 12 |
| Next.js | 3 | 4 | 5 | 12 |
| Vue | 2 | 4 | 4 | 10 |

选分数最高者，不选"最新"的，只选"最合适"的。

**关键原则：技术选型是可逆决策还是不可逆决策？** 路由框架选 React Router 还是 Next.js Router 是可逆的（都能换）。数据库选 Supabase 还是自建是半可逆的（有迁移成本）。内容存储选静态文件还是 CMS 是可逆的。

**参考案例**：Khan Academy 2012 年选 Python 是因为团队最熟悉，2021 年用户规模变大才迁移 Go——"等技术成熟了再换"是错的，但"一开始就选最复杂的"也是错的。

**教训**：选了最新的 Vite 6.x → GLIBC 版本不兼容（服务器系统太旧）→ 回退到 Vite 5.x。选技术之前先查团队环境是否支持。

# 第二章：架构设计

## 2.1 内容分层结构

**标准做法：**

内容分层分两步：

**第一步：定内容粒度（核心决策）**

回答：用户在学习时，最小的完整单元是什么？

- Khan Academy：Learning Objective（"解两步方程"），一粒米大小
- Brilliant：Lesson（一个知识点，15~30分钟），一粒葡萄大小
- StudyNotion：Subsection（一段视频，5~10分钟），一粒花生大小
- 洋葱学园：3分钟短视频，一粒芝麻大小

粒度越小，学习路径越灵活，但内容制作成本越高。

**第二步：定层级数**

| 层级数 | 适用场景 | 风险 |
|--------|----------|------|
| 2层（学科→内容） | 内容极少，初期探索 | 太粗，导航困难 |
| 3层（学科→模块→内容） | 内容中等，推荐 | 平衡较好 |
| 4~5层 | 内容庞大（视频课、知识点极多） | 维护成本高，导航复杂 |

**标准原则：层级数由内容规模决定，不要过度设计。** 高中物理有42个模型，每个模型有六段式内容，选3层（学科→模型→六段式）是合理的。

**参考案例**：Khan Academy 用五层（Domain→Subject→Topic→Objective→Exercise），适合10万+知识点的超大型内容库；Brilliant 用四层（Path→Course→Lesson→Step），适合精而深的内容；StudyNotion 用五层（User→Course→Section→Subsection→Content），适合视频课程平台。

**教训**：粒度太粗（只有学科）→ 导航找不到入口；粒度太细（六段式每节都独立）→ 维护成本爆炸，内容更新困难。

## 2.2 数据模型设计

**标准做法：**

数据模型设计分三步：

**第一步：识别核心实体**

典型学习平台的核心实体：

- **用户**（User）
- **内容单元**（Content / Course / Model）
- **进度**（Progress）
- **练习/测验**（Exercise / Question）
- **关系**（Content 之间有没有前置依赖？）

**第二步：定义实体之间的关系**

常见关系：
- 用户 → 内容：学习（一个用户学多个内容）
- 内容 → 内容：前置依赖（学完A才能学B）
- 内容 → 练习：包含（一个模型对应多道题）
- 用户 → 进度：追踪（一个用户有多个内容的进度）

**第三步：写出接口/Schema**

每个实体写出：字段名、类型、是否必填、默认值、业务含义。

```typescript
interface Question {
  id: string
  subject: 'physics' | 'math' | 'chemistry' | 'biology' | 'chinese' | 'english'  // 学科标识，预留六科扩展
  contentId: string         // 关联的内容单元
  difficulty: 'B' | 'J' | 'T'
  type: '选择题' | '填空题' | '计算题'
  question: string          // Markdown + LaTeX
  options: string[] | null  // 选择题有，填空/计算题无
  answer: string
  explanation: string       // Markdown + LaTeX
  points: string[]          // 关联知识点（用于后续拓展搜索）
  prerequisiteQuestionIds?: string[]  // 前置题目（可选）
}
```

**多学科扩展原则**：所有数据模型必须包含 `subject` 字段，所有页面组件禁止硬编码学科名称，必须通过路由参数或配置注入。

**参考案例**：StudyNotion 的 User Schema 包含 progress 数组（嵌套 Course→Section→Subsection 进度）；Khan Academy 的 Topic Graph 用 prerequisite/postrequisite 边表示知识点之间的关系。

**教训**：两套题库并存（一套在内容文件里，一套在题库系统里），数据类型完全不同（`{type, q, a}` vs `{options, answer}`）→ 用户困惑，维护成本高。发现后强制统一到题库系统。

## 2.3 技术架构文档

**标准做法：**

技术架构文档必须包含四个部分，任何人读了都能照着做：

1. **系统概览**：系统做什么、有哪些模块、模块之间什么关系
2. **数据流图**：内容怎么进入系统、用户数据怎么存储和读取
3. **接口定义**：核心数据模型（用 TypeScript / JSON Schema / OpenAPI）
4. **部署方式**：怎么构建、怎么部署、依赖什么环境

**标准格式**：Markdown + 图（系统架构图/数据流图），存放在代码仓库根目录，随代码一起版本管理。

**文档评审标准**：写完后让一个不了解项目的人读，他能不能照着做出一个最小可用的原型？能 → 文档及格；不能 → 重新写。

**参考案例**：StudyNotion 的 README 包含完整的 Mongoose Schema、API 端点列表、目录结构说明，任何人都能照着搭建；Khan Academy 在官方博客公开了2021年架构重构的决策过程和理由。

**教训**：没有架构文档 → 新人无从下手（只能靠口口相传）；架构文档和实际代码不一致 → 文档变成误导。

# 第三章：开发准备

## 3.1 项目初始化

**标准做法：**

项目初始化必须按顺序完成以下步骤，每步完成后验证再进入下一步：

**第一步：脚手架生成**

```bash
npm create vite@latest my-app -- --template react-ts
cd my-app && npm install
```

生成后运行 `npm run dev` 验证浏览器可打开。

**第二步：目录结构约定**

在项目根目录下建立固定的目录结构，全员遵守：

```
src/
  data/          # 静态数据文件（每个内容模块一个 .ts）
  sections/      # 页面组件（每个路由对应一个文件）
  components/    # 共享组件（可在多个页面复用）
  stores/        # Zustand 状态管理
  lib/           # 工具函数（supabase配置等）
  types/         # TypeScript 类型定义
scripts/         # 工具脚本（审计、构建辅助）
```

**第三步：基础依赖安装**

```bash
npm install tailwindcss @tailwindcss/vite
npm install react-router-dom
npm install @supabase/supabase-js
npm install zustand
npm install react-markdown katex
```

安装后立即运行 `npm run build` 确认无错误。

**第四步：规范约定**

- commit 格式：中文，描述改动内容（例：`feat: 添加题库答题页`）
- 分支命名：`feature/功能名` / `fix/问题名`
- 代码风格：ESLint + Prettier，commit 前自动检查

**参考案例**：Vite 官方脚手架内置了 TypeScript + React + Tailwind 支持，可一键生成；StudyNotion 的 GitHub README 有完整的项目初始化步骤。

**教训**：跳过"脚手架验证" → 后期构建失败不知道是依赖问题还是代码问题；目录结构不约定 → 各人按各人习惯放文件，后来者找不到代码。

## 3.2 组件架构

**标准做法：**

组件开发遵循"共享组件先行"原则：

**第一阶段：共享组件（所有人用）**

| 组件 | 职责 |
|------|------|
| AppLayout | 统一页面布局（顶部导航+左侧学科导航） |
| MarkdownRenderer | 内容渲染（文字+公式） |
| BackButton | 返回导航 |
| SubjectTemplate | 学科页面模板（接收 subject 参数渲染不同学科） |

**第二阶段：页面组件（依赖共享组件）**

每个路由对应一个页面组件，放在 `sections/` 目录下。

**第三阶段：数据层（被页面组件调用）**

数据获取逻辑放在 `stores/` 或直接调用 Supabase，页面组件不直接写数据请求。

```
页面组件（sections/）→ 共享组件（components/）→ 布局
         ↓
    数据请求（stores/）
```

**关键原则：页面组件之间不允许直接互相调用，只通过路由跳转。** 页面 A 要跳到页面 B → 用 `<Link to="...">` 而不是直接 import 页面 A 的函数。

**多学科扩展原则**：所有学科页面必须复用 `SubjectTemplate`，禁止为每个学科复制一份页面代码。学科差异通过配置文件（`subject.config.ts`）注入。

**参考案例**：Brilliant 每个 Lesson 有固定布局（概念区在左，练习区在右），全局统一；StudyNotion 有统一的 Sidebar 导航组件，所有页面共享。

**教训**：先做页面后补共享组件 → AppLayout 在多个文件中重复定义，改一次要改五个文件。

## 3.3 状态管理

**标准做法：**

状态分层管理，不同场景用不同方案：

| 状态类型 | 存放位置 | 说明 |
|----------|----------|------|
| 全局用户状态（是否登录） | Zustand store | 多个组件共享 |
| 学习进度 | Zustand store + Supabase | 本地优先写，异步同步云端 |
| 收藏夹 | Zustand store + Supabase | 同上 |
| 组件内部 UI 状态 | useState | 不需要共享的状态 |
| 跨层级共享（theme等） | React Context | 小量数据 |

**什么时候加新 Zustand store**：当某个功能需要跨多个组件共享数据，且这些数据需要持久化（刷新页面不丢失）时。

**什么时候不加**：只在两个组件之间共享 → 用 props；只是页面内状态 → 用 useState。

**参考案例**：StudyNotion 用 Redux Toolkit 管理课程进度（所有页面共享），体系完整但较重；Brilliant 用 React 原生状态管理（因为场景简单），代码轻量。

**教训**：过度设计 → 所有状态都上 Zustand → 调试时要看五六层 store 才知道某个值从哪来；正确做法是"够用就行，逐步升级"。

## 3.4 移动端与性能基准

**标准做法：**

学习平台面向高中生，移动端（手机、平板）使用场景占比极高，必须在开发初期确立移动端标准：

**响应式断点**

| 断点名 | 宽度范围 | 布局调整 |
|--------|----------|----------|
| mobile | < 640px | 单列、底部导航、触控区域 ≥ 44px |
| tablet | 640px ~ 1024px | 双列或侧边栏折叠 |
| desktop | > 1024px | 完整三栏布局 |

**性能基准（上线前必须达标）**

| 指标 | 目标值 | 检测方式 |
|------|--------|----------|
| 首屏加载时间（FCP） | < 1.5s | Lighthouse |
| 可交互时间（TTI） | < 3.5s | Lighthouse |
| 累计布局偏移（CLS） | < 0.1 | Lighthouse |
| 总阻塞时间（TBT） | < 200ms | Lighthouse |
| Lighthouse 总分 | > 90 | Lighthouse CI |

**触控体验标准**
- 按钮/选项最小触控区域：44 × 44 px
- 公式缩放：支持双指缩放，禁止固定 viewport 导致无法缩放
- 字体基准：16px（禁止小于 12px，防止移动端浏览器强制缩放）

**参考案例**：Khan Academy 移动端占比超 60%，其响应式方案以触控优先；Brilliant 的移动端 Lesson 采用底部固定操作栏，单手操作友好。

**教训**：桌面端开发完成后才适配移动端 → 布局重构成本翻倍，部分组件需要重写而非简单调整 CSS。

# 第四章：内容建设

## 4.1 内容存储策略

**标准做法：**

根据内容的变更频率和访问模式选择存储方式：

| 内容类型 | 存储方式 | 说明 |
|----------|----------|------|
| 模型内容（六段式文本） | TypeScript 文件（`/data/*.ts`） | 变更少，版本化管理，随代码发布 |
| 题库数据 | TypeScript 文件 + JSON 源文件 | 变更中等，批量导入有脚本 |
| 用户数据（进度/收藏） | Supabase | 需要持久化和跨设备同步 |
| 用户生成内容（笔记） | Supabase | 需要云端同步 |
| 配置/开关 | 环境变量或 `constants.ts` | 少量，非业务数据 |

**不要自建 CMS**：CMS 的维护成本（数据库、Web 服务器、内容审核流程）远超过它带来的价值。初期用静态文件足够。

**内容编辑者和开发者的协作方式**：内容团队提供 JSON 或 Markdown，开发者写转换脚本导入，审核后合并到代码库。不需要内容团队会写代码。

**参考案例**：Khan Academy 有专门的内部内容管理工具；Brilliant 全部内容静态化，发布时直接重新构建；StudyNotion 用 CMS + MongoDB（内容团队通过管理后台编辑）。

**教训**：过早引入 CMS（自建内容管理后台） → 复杂度上升，内容团队不会用 → CMS 变成摆设。

## 4.2 内容生产接口关系

**标准做法：**

本平台的内容生产流程为 **AI 生成 → 开发者自检 → 数据转换 → 部署**，无全自动化审核环节。流程设计以"机器可处理、格式可校验、错误可回滚"为核心。

**内容生产流水线**

```
AI 生成原始内容（Markdown / JSON）
    ↓
开发者自检（人工检查格式和内容质量）
    ↓
数据转换脚本 → 生成 /data/*.ts 或 /data/*.json
    ↓
构建验证（npm run build）--- 失败 → 退回检查数据格式
    ↓
截图验证 + Console 检查
    ↓
合并到主分支 → 触发部署
```

**说明**：当前阶段不要求自动化校验脚本，由开发者负责自检。随着内容规模扩大，可逐步引入自动化门禁。

**接口定义（AI 产出物标准）**

AI 生成的内容必须满足以下接口，才能被后续流程消费：

```typescript
// AI 产出物接口
interface AIGeneratedContent {
  subject: string              // 学科标识，枚举值
  modelId: string              // 模型编号，如 'PHY-M01'
  version: string              // 内容版本，如 'v1.0'
  segments: {
    type: '概念' | '公式' | '例题' | '图解' | '应用' | '总结'
    content: string             // Markdown + LaTeX
    images?: string[]           // 图片 URL 列表
  }[]
  questions: {
    id: string
    difficulty: 'B' | 'J' | 'T'
    type: '选择题' | '填空题' | '计算题'
    question: string
    options?: string[]
    answer: string
    explanation: string
  }[]
}
```

**开发者自检清单（人工检查项）**

| 检查项 | 通过标准 | 失败处理 |
|--------|----------|----------|
| LaTeX 语法 | 所有 `$...$` 和 `$$...$$` 成对闭合 | 标记错误位置，退回修复 |
| Markdown 结构 | 标题层级连续（# → ## → ###），无空标题 | 修复或退回 |
| 题目完整性 | 选择题必须有 options，填空/计算题 options 为 null | 退回修复 |
| 学科字段有效性 | subject 必须在枚举范围内 | 拒绝入库 |
| 图片 URL 可访问性 | HTTP 200 或使用已托管资源 | 使用占位图或退回 |

**回滚策略**

- 每次内容更新生成版本号（`content-v{日期}-{序号}`）
- 构建失败时自动回退到上一版本内容
- 保留最近 10 个版本的数据快照，支持一键回滚

**参考案例**：Brilliant 的内容生产高度自动化，编辑只负责最终质量把关；Khan Academy 早期内容由教师手动录入，后期才建立自动化流水线。

**教训**：AI 生成内容不经过格式校验直接入库 → LaTeX 语法错误导致构建失败，且难以定位是哪一道题的问题 → 必须建立"生成即自检"的门禁。

## 4.3 内容质量保障

**标准做法：**

内容上线前必须检查：

1. **公式渲染**：所有 LaTeX 公式在目标环境能正确显示（包括字体加载）
2. **图片加载**：所有图片 URL 可访问（非相对路径）
3. **链接有效性**：内部链接跳转到正确的页面，不跳404
4. **内容格式**：Markdown 语法正确，无未闭合的标签

**公式渲染标准做法**（以 KaTeX 为例）：

- 字体本地化：把 KaTeX 字体文件（woff2）放在 `public/fonts/` 目录，CSS 引用 `/fonts/` 路径
- KaTeX JS 从 CDN 加载时做好容错：加载失败显示原文 LaTeX，不崩溃
- CSS 使用 `font-display: block` 防止字体加载时文字跳动

**内容审核流程**：

```
AI 生成 → 开发者自审 → 数据转换 → 截图验证 → 合并到主分支 → 部署
```

每个模块首次上线必须有截图验证，之后增量更新可依赖自动化检查。

**参考案例**：Khan Academy 的练习题有专门的题目质量审核流程（难度分级、内容审查）；Brilliant 每个 Lesson 有编辑审校。

**教训**：公式渲染依赖 jsdelivr CDN → 部分网络 DNS 失败 → 公式不显示 → 降级方案：字体本地化 + 容错处理，公式无法渲染时显示原文 LaTeX，不崩溃。

## 4.4 题库系统设计

**标准做法：**

题库系统是学习平台的核心功能，设计要点：

**1. Question 接口统一**

所有题目使用统一的 Question 接口，不因学科/模型不同而变化：

```typescript
interface Question {
  id: string              // 唯一标识（'PHY-B01'）
  subject: string         // 学科（'physics' | 'math' | ...）
  contentId: string        // 关联内容（'PHY-M04'）
  difficulty: 'B' | 'J' | 'T'  // 基础/进阶/挑战
  type: '选择题' | '填空题' | '计算题'
  question: string        // Markdown + LaTeX
  options: string[] | null  // 选择题有选项，填空/计算为 null
  answer: string          // 答案
  explanation: string    // 解析（Markdown + LaTeX）
  hint: string | null     // 可选提示
  tags: string[]         // 关联知识点标签
  estimatedMinutes: number
}
```

**2. 按难度分层展示**

题库列表按 B/J/T 三层难度分区展示，用户自选难度进入：

- **基础（B）**：概念理解题，正确率应 > 70%
- **进阶（J）**：综合应用题，正确率应 50%~70%
- **挑战（T）**：综合/创新题，正确率应 30%~50%

**3. 题目与内容关联**

题目在模型详情页显示（"本节练习"区块），也在独立题库页显示。两者数据源相同（同一个 `questionsByModel` 对象），不要有两套数据。

**4. 题目与套路/模型关联**

每道题通过 `routineIds` 和 `modelId` 关联到对应套路和模型，支持"做完这道题 → 查看相关套路"的跳转路径。

**5. 答案提交后显示反馈**

提交后立刻显示：
- 正确/错误状态（选项边框变色）
- 正确答案（高亮）
- 解析（展开）
- 下一题按钮

**参考案例**：Khan Academy 的 Exercise 接口支持变体题生成（同一模板生成不同参数）；Brilliant 每道题配套即时反馈（答对显示绿色，答错显示红色+解释）。

**教训**：两套题库并存（一套在模型内容文件里的自测题，一套在题库系统的 Question 对象）→ 接口完全不同、数据不连通 → 强制统一，删除冗余的自测题代码。

# 第五章：测试与质量

## 5.1 开发自测流程

**标准做法：**

每次代码改动后，必须按以下顺序验证：

**第一步：构建检查**

```bash
npm run build
# 必须输出 "built in X.XXs"
# 必须 EXIT code = 0
```

构建检查清单：
- [ ] 无 TypeScript 错误
- [ ] 无 Rollup/Vite 错误
- [ ] dist/ 目录下有 index.html 和 assets/ 目录
- [ ] EXIT code = 0

**第二步：截图验证**

打开部署后的 URL（强制刷新 Ctrl+Shift+R），截图确认：
- 页面正常加载（非白屏）
- 无 Console 错误（Console 面板无红色）
- 主要功能可见（导航正常、内容显示）

**第三步：功能验证**

针对改动的功能点进行专项验证：
- 改动了题库 → 测试题库列表和做题页
- 改动了模型 → 测试模型详情页
- 改动了路由 → 测试新路由和旧路由是否正确跳转

**交付标准**：截图 + 构建 EXIT 码 = 0，是部署前的最低门槛。

**参考案例**：GitHub Actions CI 自动运行构建 + 单元测试，合并不符合条件的 PR；Khan Academy 有专门的练习题质量审核流程。

**教训**：跳过截图验证 → 发布后用户看到白屏才知道有问题；"构建成功"不等于"运行正常"——必须看浏览器。

## 5.2 代码审计

**标准做法：**

代码审计分自动化和人工两类：

**自动化审计（脚本检查）**

每次构建前自动运行，覆盖常见问题：

| 检查项 | 什么情况算问题 | 修复方式 |
|--------|--------------|----------|
| 死代码 | 变量被计算但未使用 | 删除计算代码 |
| 重复 import | 同一个模块被 import 了两次 | 合并为一次 import |
| 类型错误 | TypeScript 报错 | 修复类型定义 |
| 遗留引用 | 应被替换的文件仍被引用 | 更新引用路径 |
| 重复数据 | 两套数据描述同一个实体 | 合并数据源 |

审计脚本放在 `scripts/audit.sh`，构建前自动运行，EXIT ≠ 0 时阻止部署。

**人工审查时机**：

- 新功能首次开发：审查数据模型设计是否合理
- 跨模块改动：审查改动是否影响其他模块
- 重构：审查是否引入了新的复杂度

**参考案例**：前端团队用 ESLint + Prettier 做自动化检查；大型项目用 SonarQube 做代码质量分析。

**教训**：没有自动化检查 → `physicsModels` 从 `physicsData.ts` 导入的遗留问题在 4 个文件中存在了两周才发现。

## 5.3 部署前验证

**标准做法：**

部署到 CDN 后，必须完成以下步骤才算部署完成：

**步骤1：截图**

打开新 URL，截图覆盖：
- 首页（确认整体样式正常）
- 本次改动的核心页面（如题库、模型详情）
- 移动端视图（浏览器 DevTools 切换手机模式）

**步骤2：Console 检查**

打开浏览器 DevTools → Console 面板，检查：
- 无红色错误（Error 级别）
- 无网络请求失败（Network 面板 404/500）

**步骤3：新旧 URL 切换**

将新 URL 告知用户，确认旧 URL 仍然可用（不删除，作为回退）。

**步骤4：更新记录**

在部署记录中写入：

```
版本 | URL | 时间 | 改动内容 | 验证人
```

**参考案例**：Vercel/Netlify 自动做部署预览（Preview URL），团队成员可以在合并前看到实际效果。

**教训**：URL 含有数字 0 和字母 l（如 `5j0l0c52`）→ 用户分不清，访问错误的 URL 得到 404 → 选 URL 时避免数字 0 和字母 l 的组合。

# 第六章：部署与迭代

## 6.1 构建与发布

**标准做法：**

**构建优化**：

```bash
# 生产构建
npm run build

# 关键优化项
# - code splitting：按路由懒加载
# - tree shaking：移除未使用的代码
# - chunk size：单个 JS 文件 < 500KB
```

chunk size 超过警告阈值时，调整 `vite.config.ts`：

```typescript
build: {
  rollupOptions: {
    output: {
      manualChunks: {
        vendor: ['react', 'react-dom', 'react-router-dom'],
        katex: ['katex'],
      }
    }
  }
}
```

**部署策略：CDN 测试验证 → 正式发布平台**

| 阶段 | 部署目标 | 用途 | 保留策略 |
|------|----------|------|----------|
| 开发测试 | 本地 / 局域网 | 开发调试 | 随时重建 |
| 内测验证 | CDN（临时 URL） | 功能验证、截图确认、移动端测试 | 保留至正式发布后 7 天 |
| 正式发布 | 选定平台（GitHub Pages / Vercel / 自有服务器） | 用户访问 | 长期保留，版本回退 |

**CDN 测试验证标准**：
- 每次构建产物（`dist/`）上传 CDN 获得独立 URL
- 测试 URL 用于团队内部验证，不对外公开
- 验证通过（截图 + Console 检查 + 功能测试）后，再推送至正式发布平台
- 正式发布平台选定后，测试 CDN URL 保留 7 天作为回退

**URL 管理**：

- 每次部署使用新 URL（平台自动生成）
- 旧 URL 保留（作为回退）
- 当前稳定 URL 记录在 MEMORY.md 中

**参考案例**：Khan Academy 2021 年用蓝绿部署：新版本先部署到部分服务器，验证通过后切换全部流量；StudyNotion 用 Vercel 自动部署，每次 PR 都有预览 URL。

**教训**：构建产物（dist/）没有更新就部署 → "修复"无效，用户看到的还是旧代码 → 每次构建后检查 dist/ 时间戳。

## 6.2 线上监控与健康检查

**标准做法：**

正式上线后，必须建立最低限度的线上监控：

**错误日志人工巡检**

- 每天/每周查看一次浏览器控制台错误报告（通过用户反馈或主动查询）
- 记录错误频率和类型：构建错误、网络错误、运行时错误、第三方服务错误
- 错误率（用户遇到错误的比例）> 5% 时需立即排查

**性能巡检**

- 每月运行一次 Lighthouse 全量扫描
- 监控指标：FCP、TTI、CLS 任一指标劣化 > 20% 时排查
- 资源监控：JS bundle 总大小增长 > 20% 需说明原因

**数据安全与备份**

- Supabase 数据库开启自动备份（每日）
- 用户敏感数据（学习进度、收藏）传输使用 HTTPS
- 如面向未成年用户，需遵守《未成年人网络保护条例》数据收集最小化原则

**说明**：暂未接入 Sentry/Bugsnag 等自动化错误监控服务，当前以人工巡检为主。内容规模扩大后建议逐步引入自动化监控。

**参考案例**：Brilliant 使用 Datadog 监控用户学习路径中的异常中断点；Khan Academy 有专门的"学习体验健康度"仪表盘。

**教训**：没有线上监控 → 用户报告"页面打不开"时无法复现，只能盲猜；没有备份 → 一次误操作导致用户学习进度丢失，无法恢复。

## 6.3 架构健康检查

**标准做法：**

每季度做一次架构健康检查，流程：

1. **回顾**：这季度做了什么改动？架构上有哪些债务？
2. **对比**：和参考案例相比，我们缺什么？有什么不对的地方？
3. **优先级**：列出问题，按"影响范围 × 修复成本"排序
4. **计划**：下一季度最多做 3 个架构改动，其他靠维护解决

**架构问题识别方法——参考案例对比法**：

| 参考案例 | 他们的做法 | 我们有没有 | 优先级 |
|----------|-----------|-----------|--------|
| Khan Academy | Topic Graph 前置依赖 | 无 | 高 |
| Khan Academy | 掌握度追踪（5级） | 只有完成/未完成 | 中 |
| Brilliant | 讲练合一（内嵌练习） | 完全分离 | 中 |
| StudyNotion | Section 级别进度 | 模型级别 | 中 |

**小步迭代原则**：每次改动不超过一个模块，改动后验证再继续。不追求一次解决所有问题。

**参考案例**：Khan Academy 2021 年的架构重构花了两年，不是一口气换完，而是分阶段：先把认证服务拆出来，再迁数据库，最后迁业务逻辑。

**教训**：13 个架构问题积累 → 系统审计发现全貌 → 优先级排序后分批修复（高优先级的4个已修复，剩余在后续迭代中逐步处理）。
